SIEM Integrations

Forward OT security logs to your SIEM platform with pre-built collectors, correlation rules, and dashboards.

Architecture Overview

All OT lab events flow through Redis pub/sub channels and are forwarded to your SIEM platform by dedicated collectors.

Supported Platforms

Five production-ready collectors with protocol-aware parsing and MITRE ATT&CK enrichment.

Sumo Logic

HTTP Source — JSON Log Forwarding • Pre-built OT Dashboards

HTTP Source collector with JSON log forwarding and pre-built OT dashboards. Events are batched and flushed to your Sumo Logic HTTP endpoint with full field extraction and MITRE ATT&CK technique enrichment.

Log Format

JSON via HTTP POST

Configuration

SUMO_ENDPOINT=https://endpoint1.collection.us2.sumologic.com/receiver/v1/http/...

Features

  • Pre-built OT security dashboards
  • MITRE ATT&CK enrichment
  • Automated parsing
  • Batch flushing (100 events or 10 seconds)

Splunk

HEC — CIM-Compliant Field Mapping • OT Security App

HEC collector with CIM-compliant field mapping and OT security app. Events are forwarded via HTTP Event Collector with automatic sourcetype assignment and index-time field extraction for fast search performance.

Log Format

JSON via HEC (HTTP Event Collector)

Configuration

SPLUNK_HEC_URL=https://splunk.example.com:8088/services/collector
SPLUNK_HEC_TOKEN=your-hec-token-here

Features

  • CIM field mapping
  • sourcetype=vulnot:ot
  • Pre-built correlation searches
  • Index-time field extraction

ELK Stack

Elasticsearch — Kibana Dashboards • Logstash Pipeline Configs

Elasticsearch ingest with Kibana dashboards and Logstash pipeline configs. Events are bulk-indexed with OT-specific field mappings, lifecycle policies, and ready-to-import Kibana dashboard exports.

Log Format

JSON via Elasticsearch bulk API

Configuration

ELK_URL=https://elasticsearch.example.com:9200

Features

  • Index template with OT field mappings
  • Kibana dashboard exports
  • Logstash pipeline configs
  • ILM policy for log retention

Google Chronicle

UDM — Unified Data Model Ingestion • Chronicle Detection Rules

UDM-formatted ingestion with Chronicle detection rules. OT events are mapped to the Unified Data Model with entity context, enabling native Chronicle detection rules written in YARA-L for cross-source correlation.

Log Format

UDM (Unified Data Model)

Configuration

CHRONICLE_CUSTOMER_ID=your-customer-id
GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account.json

Features

  • UDM entity mapping
  • Chronicle detection rules (YARA-L)
  • Entity analytics
  • Threat intelligence integration

Syslog / CEF

RFC 5424 • RFC 3164 • CEF Format — UDP / TCP / TLS

RFC 5424, RFC 3164, and CEF formats over UDP, TCP, or TLS. The universal collector for any syslog-compatible receiver, with OT-specific CEF extensions and configurable facility/severity mapping.

Log Format

Syslog (RFC 5424/3164) or CEF

Configuration

SYSLOG_HOST=syslog.example.com
SYSLOG_PORT=514
SYSLOG_PROTOCOL=tcp   # udp | tcp | tls

Features

  • UDP/TCP/TLS transport
  • CEF format with OT extensions
  • Facility/severity mapping
  • Compatible with any syslog receiver

Log Channels

Each collector subscribes to one or more Redis pub/sub channels carrying structured OT event data.

Channel Description Events/sec Format
ot_attacks Attack detection alerts ~10-50 JSON
ot_network Network traffic metadata ~100-500 JSON
ot_process Process value changes ~50-200 JSON
ot_compliance Compliance check results ~1-5 JSON
ot_forensics Forensic evidence events ~5-20 JSON

All collectors subscribe to Redis pub/sub channels and support batch flushing, MITRE ATT&CK enrichment, and multi-protocol parsing.
